## The sample_tick pattern (clock-enable, not a separate clock)

Every module in this pipeline runs off the same fast system clock (clk, 100MHz by default) but only actually does anything on
cycles where sample_tick is high — a single-cycle pulse generated by ecg_rom's internal clock divider, firing at the target
ECG sample rate (200Hz).

This is a clock-enable pattern, not a genuine second clock domain, there's only one clock in the whole design, sample_tick 
just gates when the logic inside each always @(posedge clk) block actually updates.

Why this matters if asked: this avoids the complexity and metastability risk of a true multi-clock-domain design (which 
would need proper synchronizers at every crossing). It's the right choice here because everything derives from one clock
but if this design were extended to accept a live ADC with its own independent clock, that boundary would need a real 
synchronizer (e.g. a 2-flop synchronizer on the ADC's data-ready signal), not just a clock-enable pulse.

## The non-blocking assignment "one-cycle lag" pattern — appears everywhere

This shows up in almost every filter stage and caused real confusion early in development, so it's worth having a crisp
mental model. In Verilog, every <= (non-blocking) assignment in an always block
reads pre-edge (i.e. "old") register values — including on the right-hand side of other assignments in the same block.
So y1 <= lpf_out actually captures lpf_out's value from the previous cycle, not the value being computed in this same cycle.
This is correct, standard, intentional hardware behavior (it's literally how registers work — a value can't be read and 
written in the same instant) — but it means:

Every filter stage has an inherent one-sample_tick-cycle pipeline latency baked in — the effect of a given input sample 
doesn't appear in that stage's output until the next sample_tick, not the current one.
When hand-deriving expected testbench values or reading waveforms, account for this lag or you'll misread the trace as 
wrong when it's actually correct (this happened multiple times during development see 03-bugs-and-fixes.md, the derivative 
testbench first draft).

When building a reference model in a testbench (or in Python/MATLAB) to check a DUT, this lag must be replicated exactly, 
or the reference will disagree with a correct DUT.

## Bit-width growth through the pipeline — why each stage is wider than the last

Every stage grows its output width relative to its input, to avoid overflow from that stage's own gain:

| Stage | Output width (relative to data_width, default 16) | Why |
| --- | --- | --- |
| ecg_rom | data_width (16 bits) | Raw sample, no processing | 
| lpf | data_width+6 (22 bits) | ~36x DC gain (derived from the filter's transfer function at DC: H(1) = 6² = 36, since it's the square of a 6-tap boxcar) | 
| hpf | data_width+8 (24 bits) | Small additional headroom over lpf's output | 
| derivative | data_width+8 (24 bits) | Same order of magnitude as hpf | 
| squaring | 2*data_width+16 (48 bits) |  Squaring roughly doubles the bit count needed |
| mwi  2*data_width+16 (48 bits), internal sum uses +5 more bits | Summing 32 (2⁵) terms needs 5 extra headroom bits |

Why this matters if asked: undersizing any of these would cause silent overflow/wraparound — exactly the class of bug found
in this project's lpf/hpf (see 03-bugs-and-fixes.md), though that particular case turned out to be a numerical-stability 
issue rather than simple undersizing. Always be able to justify a declared width from the actual worst-case gain of the stage
, not just "seemed wide enough."

## Parameterization philosophy

All filter-chain modules (lpf, hpf, derivative, squaring) take the same base data_width parameter (default 16) and derive 
their own output widths internally using fixed offsets (+6, +8, etc.), so changing data_width at the top level automatically 
and correctly resizes every downstream stage without manual recalculation. mwi and peak_detect instead take an explicit width
parameter matching the previous stage's actual output width, since their gain isn't a fixed offset from data_width (it 
depends on N, the window size).

ecg_top.v wires everything together using localparam-derived widths (LPF_W, HPF_W, SQ_W) computed from the single top-level
data_width, so the whole pipeline stays consistent if that parameter changes.

## Reset style

Every module uses an asynchronous reset (posedge clk or posedge rst) —reset takes effect immediately regardless of clock edge
which is standard for FPGA designs where a global reset needs to clear state quickly at power-up, independent of clock timing.
